## 1. Current "UI" Audit

### What exists today

This repo has no traditional frontend, but it has well-established
**document UI patterns**:

| Surface | Where | What it teaches us |
|---|---|---|
| Plan documents | `docs/plans/*.md` | Metadata header, numbered sections, ASCII tables, principle-citation style |
| Generated reports | `scripts/memory-health.mjs` output | HTML-comment marker on first line, status-badge frontmatter, metric tables, decision-rule block, optional `--out <path>` write target |
| Skill definitions | `skills/*/SKILL.md` | Phase-numbered structure, references-table footer for progressive disclosure |
| Project context | `AGENTS.md`, `CLAUDE.md` | Stable section anchors, env-var tables, tradeoff registers |
| GH issue body (sticky) | memory-health workflow output | Renders the same markdown that `--out` produces; auto-closes with a comment |

### Existing visual / structural conventions

- **Tables for structured data** (no fancy charts) — render in every
  surface (GitHub, VS Code, Obsidian, terminal as ASCII).
- **HTML comment markers** as sticky-issue identifiers (e.g.
  `<!-- audit-loop:memory-health -->`) — proven pattern; reuse.
- **Code fences with language hints** for syntax highlighting on GitHub.
- **No emojis** in checked-in docs (per project style; `Only use emojis if the user explicitly requests it`).
- **Stable anchor links** auto-generated by GitHub from headings —
  `## My Heading` → `#my-heading`. We rely on this rather than
  hand-maintaining anchors, so heading text becomes part of our public API.

### Pain points we're addressing

- **No queryable map of the codebase**: humans can't easily ask "what
  does this function do, and what else does similar work?"
- **Drift is invisible until something breaks**: duplicate functions get
  written because nobody scrolls through a 5000-symbol catalogue at plan time.
- **Audit findings don't relate to architecture**: a HIGH duplication
  finding currently reads as a flat assertion; humans can't see the
  cluster it belongs to.

### Reusable patterns

- Mermaid blocks render natively in GitHub PRs, VS Code preview,
  Obsidian, Claude Code (markdown render). No plugins.
- The `<details><summary>` collapsible block renders everywhere we care
  about. Used for long-tail items.
- The memory-health markdown structure (status badge, metric table,
  decision rule, samples) is the proven shape for periodic-report docs.

## 2. User Flow & Wireframe

### Three primary "user flows"

**Flow A — Reviewing a PR that touches the codebase**:
GitHub → PR → Files changed → diff includes `docs/architecture-map.md` →
reviewer scans the diff to see what architectural impact the change had
(new symbol added, duplication cluster appeared, layering violation
introduced).

**Flow B — Reading a plan output before implementation**:
Human runs `/plan-backend add X` → plan output includes a
"Neighbourhood considered" callout near the top → human scans the
near-duplicates the planner found → reviews recommendation tags
(reuse / extend / justify-divergence) → either accepts the plan or
pushes back ("but we already have Y, just extend it").

**Flow C — Investigating the weekly drift-sweep issue**:
GitHub Issues → `architectural-drift` label → open sticky issue → reads
top-N highest-impact items → clicks symbol links → arrives at exact
file:line in the source.

## 3. UX Design Decisions

| Decision | UX Principles cited | Reasoning |
|---|---|---|
| **Stable anchor links via heading text** | #14 Recognition over Recall, #10 Consistency | A reviewer's PR comment can `#cellar-grid` and stay valid across regenerations. We rely on GitHub's deterministic anchor algorithm. |
| **Status badge in document header** (`Status: green/amber/red`) | #11 Feedback & System Status, #17 Visual Hierarchy | First-glance signal — reader knows whether to dig in. Mirrors memory-health's pattern. |
| **Per-domain Mermaid + table pairing** | #1 Proximity, #6 Common Region, #13 Progressive Disclosure | Diagram for big-picture, table for drilldown — both same domain in same section. Reader chooses depth. |
| **Drift cluster highlighted in both Mermaid AND table row** | #2 Similarity, #11 Feedback | Same condition signalled in two views — reader can spot it from either entry point. |
| **`<details>` collapsible long tail** | #13 Progressive Disclosure, #15 Hick's Law, #18 Whitespace | Top-N visible by default; the rest tucked away. Doesn't overwhelm; full info still 1 click away. |
| **Recommendation tag column on Neighbourhood callout** (`reuse / extend / justify-divergence`) | #14 Recognition over Recall, #15 Hick's Law | Tells reader the planner's confidence + suggested action without reading paragraphs. Three options chunk decisions. |
| **Empty-state copy** ("no near-duplicates — proceed") | #11 Feedback, #14 Recognition | Reader knows the consultation ran (not silently skipped). Distinguishes "checked, found nothing" from "didn't check". |
| **Cloud-off state** explicit hint | #16 Performance Perception (cognitive), #11 Feedback | Reader knows why architectural context is missing and how to fix it. |
| **Deterministic ordering** (alphabetical within domain, by `(domain, kind, symbol_name)`) | #10 Consistency, drift-as-signal | A regenerated map with no actual changes produces a zero-line diff — drift surfaces as a noisy diff (intentional). |
| **No emojis** | Project house style; #21 Colour Contrast (terminal-readable) | Renders identically in terminal vs rich rendering. Drift markers use text (`[DUP]`, `[VIOL]`) not emoji glyphs. |
| **Mermaid `classDef` for duplication cluster** | #2 Similarity, #21 Colour Contrast | Distinct fill (`#ffe8d8`) + stroke (`#c0392b`) — readable in both light and dark GitHub themes; degrades to a label suffix in monochrome viewers. |
| **Fallback for >50-symbol domains**: summary diagram + "see flat table below" link | #15 Hick's Law, #18 Whitespace | Mermaid gets unreadable past ~50 nodes. Summary keeps overview; table preserves access to full data. |

### Cognitive load management

- **Default first-fold view**: header + status + recent-drift callout +
  TOC. A reviewer can see "is this clean?" without scrolling.
- **Top-N rule throughout**: Drift sweep top-5 visible, rest collapsed.
  Neighbourhood callout top-5, full list at end of plan. Layering
  violations top-N, full at the end.

### Accessibility

- **Mermaid diagrams degrade gracefully**: in terminals/monochrome viewers,
  Mermaid blocks render as the source code (still readable as labelled
  graph definition).
- **Tables are screen-reader-friendly** (proper headers).
- **Anchor links in flat tables** use the symbol name as link text
  (descriptive, not "click here").
- **Colour is never the only indicator**: duplication clusters get both
  a colour AND a text marker (`[DUP]` in the table row, `(dup)` suffix
  in the Mermaid label).

### Cross-check vs Nielsen heuristics

| # | Heuristic | How addressed |
|---|---|---|
| 1 | Visibility of system status | Status badge + generated timestamp + commit SHA |
| 2 | Match between system and real world | Domain names match how humans talk about the codebase (`audit pipeline`, not `pipeline_v2`) |
| 3 | User control and freedom | All collapsibles open/close at user's choice; no auto-redirects |
| 4 | Consistency and standards | Same Markdown patterns as memory-health; same anchor rules as plan docs |
| 5 | Error prevention | Cloud-off state shows hint instead of silent skip |
| 6 | Recognition over recall | Recommendation tags on neighbourhood callout |
| 7 | Flexibility and efficiency | Power users use anchor links + symbol-index API; new users use the rendered map |
| 8 | Aesthetic and minimalist | No decoration; tables + diagrams + text; long tail collapsed |
| 9 | Help users diagnose errors | Drift sweep includes "Suggested action" per cluster |
| 10 | Help and documentation | Footer "How to regenerate" + "How to interpret" sections |

## 4. Technical Architecture

### Module diagram

```
┌────────────────────────────────────────────────────────┐
│  scripts/symbol-index/render-mermaid.mjs (CLI)          │
│  npm run arch:render                                     │
└────────────┬───────────────────────────────────────────┘
             │ reads symbol_index + drift_score via cross-skill.mjs
             ▼
┌────────────────────────────────────────────────────────┐
│  scripts/lib/arch-render.mjs                            │
│  pure functions: groupByDomain, renderMermaid,          │
│  renderSymbolTable, renderHeader, renderDriftCallout,   │
│  renderLayeringSection, renderFooter                    │
└────────────┬───────────────────────────────────────────┘
             │ emits a single markdown string
             ▼
┌────────────────────────────────────────────────────────┐
│  atomicWriteFileSync(path, markdown)                    │
│  → docs/architecture-map.md (committed)                 │
└────────────────────────────────────────────────────────┘
```

A separate render module is used by the drift workflow:

```
scripts/symbol-index/drift.mjs
   ↓ reads drift_score RPC + recent_drift_history
   ↓ uses scripts/lib/arch-render.mjs renderDriftIssue()
   ↓ writes /tmp/architectural-drift.md
   ↓ workflow uploads as artifact + posts as sticky issue body
```

And a third surface — the neighbourhood callout — is rendered inside
each /plan-* skill's output by a thin formatter:

```
node scripts/cross-skill.mjs get-neighbourhood --json '{...}'
   ↓ returns JSON records
   ↓ skill embeds via a small Markdown helper (referenced by SKILL.md)
   ↓ output is the callout block above
```

### State management

This is a render-only surface; there is no client state. The
"state" is in Supabase (per backend plan).

- **Source of truth**: `symbol_index` + `symbol_layering_violations`
  Supabase tables.
- **Read consistency**: render reads at a single point in time; if a
  refresh is mid-flight the render will catch a partial view (acceptable;
  next render reconciles).
- **Determinism**: same DB state → byte-identical Markdown. Achieved via
  stable sort keys + canonical JSON formatting where embedded.

### Event handling

N/A (no DOM events). The "events" that trigger re-render are:
- `npm run arch:render` (manual)
- `/ship` skill step (per backend plan)
- weekly GH Action (per backend plan)

### Markdown / Mermaid "CSS" (style decisions)

Mermaid `classDef` definitions are emitted at the top of every diagram:

```mermaid
classDef container fill:#f5f5f5,stroke:#333,stroke-width:2px
classDef component fill:#e8f0ff,stroke:#3178c6,color:#000
classDef symbol fill:#fff,stroke:#999,color:#444
classDef dup fill:#ffe8d8,stroke:#c0392b,stroke-width:2px,color:#000
classDef violation fill:#ffd6d6,stroke:#c0392b,stroke-width:2px,color:#000
```

These render legibly in both GitHub light and dark themes (verified
manually against GitHub's Mermaid renderer; the explicit `color:#000`
prevents low-contrast text on light fills in dark mode).

### File layout

- All renders go through `scripts/lib/arch-render.mjs` (pure functions).
- Each renderer takes data + options, returns a string. No I/O in this module.
- I/O sits in CLI scripts (`render-mermaid.mjs`, `drift.mjs`) and uses
  `atomicWriteFileSync` from `scripts/lib/file-io.mjs`.

## 6. File-Level Plan

### New files

#### `scripts/lib/arch-render.mjs`

- **Purpose**: Pure render functions. No I/O.
- **Key exports**:
  - `renderArchitectureMap({ symbols, violations, driftHistory, generatedAt, commitSha, stack }) → string`
  - `renderDriftIssue({ driftScore, threshold, clusters, violations, history, generatedAt, commitSha }) → string`
  - `renderNeighbourhoodCallout({ records, targetPaths, totalCandidates, cloudStatus }) → string`
  - `groupByDomain(symbols) → Map<domain, symbol[]>`
  - `renderMermaidContainer(domain, symbols) → string`
  - `renderSymbolTable(symbols) → string`
  - `escapeMermaidLabel(s) → string` and `escapeMarkdown(s) → string` — input sanitisation.
- **Why this file**: #27 Single Responsibility, #28 Modularity, #29 DRY
  — three surfaces share renderers (header, table, drift callout, mermaid block).
- **Why pure**: #11 Testability — golden-file tests run with no DB or LLM.

#### `scripts/symbol-index/render-mermaid.mjs`

- **Purpose**: CLI. Reads symbol_index + recent drift history via
  `cross-skill.mjs`, calls `renderArchitectureMap`, writes
  `docs/architecture-map.md` atomically.
- **Flags**: `--out <path>` (default `docs/architecture-map.md`),
  `--repo-id <uuid>`.
- **Why this file**: #28 Modularity — separated from pure renderer.

#### `scripts/lib/neighbourhood-formatter.mjs`

- **Purpose**: Tiny shim importable from skill-side scripts that formats
  the neighbourhood callout for inclusion in plan output. Wraps
  `arch-render.renderNeighbourhoodCallout`.
- **Why this file**: #29 DRY — one renderer powers callout in both
  plan-backend and plan-frontend; both skills shell out to a single
  command (per backend plan: `node scripts/cross-skill.mjs get-neighbourhood`)
  whose output already includes the formatted callout.

#### `tests/arch-render.test.mjs`

- **Purpose**: Golden-file tests for all renderers.
- **Coverage**:
  - `renderArchitectureMap` snapshot for a 3-domain fixture.
  - `renderDriftIssue` snapshot for green / amber / red cases.
  - `renderNeighbourhoodCallout` for empty / non-empty / cloud-off / >100 candidates.
  - Mermaid label escaping (special chars: `<`, `>`, `"`, `|`).
  - Table cell escaping (`|`, newlines).
  - Stable ordering: same input twice → byte-identical output.
- **Why this file**: #11 Testability; deterministic output is a hard
  requirement and golden files are the right shape.

## 7. Risk & Trade-off Register

| # | Risk / Trade-off | Mitigation |
|---|---|---|
| F1 | Mermaid renders inconsistently across surfaces (GitHub, VS Code, Obsidian, terminal) | Stick to `flowchart TB` + `classDef` (most portable). Verified in all four targets manually. Fallback: source code is readable as labelled graph definition. |
| F2 | Generated `docs/architecture-map.md` produces noisy diffs even for trivial changes | Stable ordering + minimal-diff renderer (one symbol per row, no embedded JSON). Diff = drift signal (intentional design). |
| F3 | Domain inference noise → symbols mis-grouped | Per backend plan: domain_tag is LLM-derived. v1 accepts noise; v2 ships override CLI. UX caveat: noisy domain assignments make the Mermaid less useful — but the flat table is still correct. |
| F4 | Mermaid diagrams >50 nodes become unreadable | Fallback rule: domains >50 symbols get a summary diagram (top-15 by centrality) + "see flat table below" link. Documented in §2. |
| F5 | Sticky-issue auto-close stomps on a human takeover | Same protection as memory-health: only auto-close when last comment/body is from the workflow (marker present). |
| F6 | Symbol path links break across branches (relative paths to repo root) | Use repo-relative paths (`scripts/lib/x.mjs#L42`); GitHub renders these as branch-aware links from the rendered file's branch context. Tested manually. |
| F7 | Recommendation tag (`reuse / extend / justify-divergence`) is the planner's opinion, not ground truth | Tag is a hint, not a gate. Reviewer always overrides. We track outcome — if a "reuse" tag was followed and led to a poor outcome, future audits adjust (out of scope for v1). |
| F8 | Plan output bloated by neighbourhood callout + appendix | Top-5 callout near top; full list at end. Cap of 100 candidates (configurable). Plan readers can choose depth. |
| F9 | Drift report focuses attention on duplication; reviewers may ignore layering violations | Layering violations get their own dedicated section in `architecture-map.md` AND a separate top-level header in the drift issue body. Hard to miss. |
| F10 | Dark-mode contrast for highlighted Mermaid nodes | `classDef dup` uses `color:#000` to force readable text on the orange fill in both themes. Verified. |

### Deliberately deferred

- **Interactive web view** of the architecture map — out of scope.
  The committed Markdown is the human surface.
- **Hot-link from PR comment to a diff-aware view of the map** — would
  require a custom GitHub App. Use of stable anchors makes this
  unnecessary for v1.
- **Per-symbol history view** ("who changed this and when") — `git log`
  + `git blame` already do this; adding it to the map would duplicate
  for marginal value.
- **Per-reviewer customisation of which domains to render** — out of scope.
  Map is a single canonical view.
- **i18n** of the rendered surfaces — single-locale (English).

## 8. Testing Strategy

### Visual / manual checklist

- [ ] Open generated `docs/architecture-map.md` on GitHub (light + dark
      mode); verify Mermaid renders, duplication clusters are visible.
- [ ] Open in VS Code Markdown preview; verify Mermaid renders.
- [ ] Open in Obsidian; verify Mermaid renders (Obsidian has built-in
      Mermaid support).
- [ ] Open in terminal (`cat`); verify the document is comprehensible
      as plain text (Mermaid block reads as labelled graph).
- [ ] Click a symbol link in the table; verify it navigates to the
      file at the correct line in GitHub's source viewer.
- [ ] Trigger drift sweep workflow with `workflow_dispatch`; verify
      sticky issue body matches the wireframe.
- [ ] Invoke `/plan-backend` with a contrived task that has a known
      near-duplicate; verify the Neighbourhood callout appears with
      the expected symbol.
- [ ] Invoke `/plan-backend` with `unset SUPABASE_AUDIT_URL`; verify
      the cloud-off state callout appears.

### Accessibility checks

- [ ] Document is keyboard-navigable in GitHub (Tab through links).
- [ ] Tables use proper `<th>` headers (markdown converts automatically).
- [ ] Symbol link text is descriptive (the symbol name itself).
- [ ] Duplication clusters have both a text marker AND a colour signal.
- [ ] All Mermaid `classDef` colours pass WCAG AA when rendered as
      SVG (manually checked against #fff and #0d1117 backgrounds).

### Determinism / regression

- Golden-file tests in `tests/arch-render.test.mjs` lock the exact
  Markdown output for several fixture inputs.
- Re-running render on identical state must produce byte-identical
  output. CI assertion: `git diff --exit-code docs/architecture-map.md`
  after `npm run arch:render` on the same commit.

### Responsive / cross-surface

- GitHub (web, light + dark, mobile + desktop)
- VS Code (Markdown preview)
- Obsidian (Markdown view)
- Terminal (`cat` / `less`)
- Claude Code (markdown render)

### Edge cases to verify

- Repo with one domain.
- Domain with one symbol (no Mermaid; table only).
- Domain with 60 symbols (summary Mermaid + table).
- Symbol with `|` in name (escape correctly in tables).
- Path with `<` in it (Mermaid label escape).
- Multi-line `purpose_summary` (truncate at first sentence; full text
  in tooltip-style title attribute? Markdown doesn't support — truncate
  with ellipsis + full text after the table).
